# Stored Procedure Refactor Workflow

This document codifies the workflow used for the lsp_ShpGetOrdersForTopReadyToShipGroup pilot (Row 15 in the MFC database optimization tracking sheet) and applied to subsequent procs in the bulk run. Read this end to end before starting on a new procedure. It is the single source of truth for how a refactor analysis is produced.

## Goal

For each procedure in the tracking sheet, produce a self-contained refactor folder that documents the pre-refactor state, the post-refactor state, the first principles guiding the change, the evidence supporting the cost reduction, and the validation that the refactor is safe to deploy. The deliverable is durable knowledge as much as it is a code change.

## Inputs

For every procedure, gather these before starting:

1. The tracking sheet entry (`MFC database optimization tracking sheet.xlsx`). Note the issue, the refactor plan, the priority, the status, and the headline numbers. Quote them verbatim where useful.
2. The committed DDL at `stored_procedures/<ProcName>.sql`. If a refactor (v-N+1) has already been written to the file, the prior committed version becomes Original.sql; otherwise the current file is Original.sql and a refactor draft must be produced.
3. Cross-MFC Query Store output at `diagnostics/querystore_outputs/QueryStore_<date>.xlsx`. One tab per site, one row per query_id within the proc. Filter to rows whose `objectname` matches the target proc.
4. Index DDL at `refactors/<ProcName>/IndexExtract.xlsx` or a shared file. The columns are schema_name, table_name, index_name, index_type, key_columns, included_columns, has_filter, filter_predicate, row_count, size_mb. Generated by `extract_index_ddl.sql`.
5. Live execution data: STATS IO/TIME outputs and execution plans, which I capture per proc. These come in after the analysis-side work is queued.

## Pre-Work

Before opening any editor:

1. Check git history with `git log --follow --oneline stored_procedures/<ProcName>.sql`. If a v-N+1 refactor was already committed, pull the prior version with `git show <commit>:stored_procedures/<ProcName>.sql > refactors/<ProcName>/Original.sql` and copy the current file to `Refactored.sql`. Otherwise the current file is `Original.sql` and `Refactored.sql` will be produced as part of the analysis.
2. Read the tracking sheet entry. Record the headline numbers (executions, total reads, plan count, avg duration) and the proposed fix.
3. Read the Query Store output. Aggregate per site: total executions, total reads, weighted avg reads/exec, plan variant count, distinct query_id count. Record the cross-MFC roll-up.


## Analysis.md Template

Eleven sections. Each one has a clear job. Do not merge or reorder.

1. **Procedure Name & Surface Area.** What the proc does in one line; the tables touched with brief descriptions; the indexes used (extracted from the v25 plan or equivalent); the callers if known.
2. **Overview of Performance.** Two or three short paragraphs setting the cost picture from the cross-MFC view. Headline numbers, plan instability indicators, per-site spread, dominant cost driver.
3. **Evidence of Original.** 3.1 cross-MFC Query Store table with one row per site. 3.2 STATS IO/TIME from a representative MFC run, with per-table reads aggregated and per-statement timing.
4. **Issue Identification.** Walk the DDL. List each anti-pattern with the line number it lives on and a brief description. One subsection per issue.
5. **First Principles.** One subsection per principle. Cite existing masterclass notes by bare wikilink (`[[Note Name]]`). If a refactor surfaces a principle not covered by an existing note, write a fresh masterclass note in `masterclass/` and cite it here.
6. **Refactor (commented).** The new SQL as it appears in `Refactored.sql`, with inline comments explaining each changed block. Mirror the structure of the file.
7. **Risk & Rollback.** What could go wrong, what to look for in the first 24 hours, how to revert. Include a monitoring window with specific metrics to watch.
8. **Evidence of Refactor.** STATS IO/TIME from the warm-cache run on the same data state as Section 3.2, plus the post-refactor plan summary.
9. **Comparison & Improvement.** Apples-to-apples comparison of v-N to v-N+1. Read counts per table, total reads, CPU time, elapsed time, plan operator counts. Verdict at the top of the section.
10. **Validation Checklist.** Marked pass or fail against the criteria below. Net call sentence at the end.
11. **Open Items / Future Improvements.** Three subsections. 11.1 index recommendations with full CREATE INDEX statements. 11.2 proc-level changes that are out of scope for this version but ready to apply next. 11.3 schema-level changes that need DBA blessing.

## Apples-to-Apples Evidence Protocol

These rules are non-negotiable for every comparison. Rule violations land in `tasks/lessons.md`.

- Same data state for original and refactor. Capture back to back without intervening writes to the touched tables.
- Warm cache only. Run twice, discard the cold-cache numbers.
- Non-zero result set. A 0-row or near-empty result does not exercise rowgoal optimization or the dominant cost drivers like FOR XML PATH consolidation. Stage data or capture during a busy window.
- Plan shape must match the prediction. The optimizer's index choice matters, not just the query shape. Confirm via the .sqlplan file.
- Identical row count and row identities for deterministic queries.
- Capture row counts. If the proc sets `Set NoCount On`, run with `Set NoCount Off` at the session level, or capture the result grid alongside.
- Save the .sqlplan files. They are the only way to verify operator counts and index choices.

## Common Anti-Pattern to Fix Mapping

| Anti-pattern | Fix | Reference |
|---|---|---|
| Parameter sniffing volatility | Local variable assignment, OPTION (RECOMPILE), or OPTIMIZE FOR | [[Parameter Sniffing]] |
| Correlated scalar subquery in SELECT or WHERE | CTE with GROUP BY then LEFT JOIN, or temp table for repeated lookups | [[Correlated Subqueries to CTEs]] |
| Same expensive subquery repeated across UNION branches | Materialize once into temp table, run expression once over the union | [[FOR XML PATH Consolidation]] |
| Scalar UDF in SELECT or WHERE | Inline with STRING_AGG, CTE, or APPLY | [[Scalar UDF Parallelism Barrier]] |
| LEFT JOIN with OR across different join paths | UNION of INNER JOIN branches, or temp-table materialize plus EXISTS | [[LEFT JOIN OR Anti-Pattern]] |
| Table variable joined to large table | Convert to #temp table with index | [[Table Variables vs Temp Tables]] |
| Non-SARGable predicate (function on column) | Rewrite to put column bare on left side; or filtered index | [[Non-SARGable Predicates]] |
| Catch-all WHERE with OR @Param IS NULL | OPTION (RECOMPILE), or dynamic SQL with sp_executesql | [[Parameter Sniffing]] |
| TRIM on column in WHERE | Pre-trim local variable; or persisted computed column | [[Non-SARGable Predicates]] |
| NOT IN subquery with NULL risk | Convert to NOT EXISTS | [[LEFT JOIN OR Anti-Pattern]] (related) |
| Scalar `(Select 1 ...) = 1` form | Convert to EXISTS | inline note |
| UNION when label columns prevent overlap | Convert to UNION ALL | [[UNION ALL Views]] |
| Legacy `If Object_Id('TempDb..#X') Is Not Null Drop Table #X` | Use `Drop Table If Exists #X` | inline note |

## Common Gotchas

- `OPTION (...)` clauses are statement-level. They cannot live inside a CTE branch or any subquery. When consolidating multiple SELECTs into a CTE, audit each original for OPTION clauses and decide explicitly how to translate. Either keep the hinted query as its own statement and append the others via INSERT, or move the hint to the outer SELECT (broadens its scope). Either choice is a deliberate decision that should be noted in Section 7.
- Tested body must match deployed body. Same version label can be different SQL. Diff what was tested against what is committed before drawing conclusions.
- Tracking-sheet single-site numbers can be much smaller than fleet-wide actuals. Always pull the cross-MFC view before estimating impact.
- A scalar subquery comparison `(SELECT 1 ...) = 1` is a more obscure form of EXISTS. Prefer EXISTS for readability and short-circuit evaluation.
- Filtered indexes only get used when the query's filter predicate matches the index's filter predicate exactly. A filtered index `WHERE Status = 'Ready'` will not be used by a query with `WHERE Status IN ('Ready', 'Pending')`.
- Index name preservation matters when modifying an existing index. Drop and recreate keeps the name; create-new-and-drop-old changes any plan-cache or hint references. Note explicitly which path is being recommended.

## Validation Checklist (Section 10)

Mark each pass or fail using ✅ ⚠️ ❌ or ? as appropriate. Every refactor must pass all of these before being declared done.

- **Same data state.** Captures taken back to back, ideally with a data freeze.
- **Warm cache only.** Two runs, cold-cache numbers discarded.
- **Non-zero result set.** Both runs returned at least one row.
- **Identical result set.** Same row count and same row identities for deterministic queries.
- **Plan shape matches prediction.** The optimizer used the predicted indexes. Confirmed against .sqlplan.
- **No new error or warning messages.** No cardinality errors, conversion warnings, or excessive memory grant warnings.
- **Warm-cache elapsed time at or below original.** Wall clock has not regressed.

End the section with a one-sentence net call.

## Prose Conventions

These apply to all deliverables: Analysis.md, Refactored.sql comments, masterclass notes, and `tasks/lessons.md` entries.

- Plain prose. Sentences are sentences.
- No em-dashes. Use commas, periods, colons, or rewrite the sentence to flow without one.
- No clipped phrasing or rhetorical flourishes.
- No emojis except in validation checklist status markers (✅ ⚠️ ❌ ?).
- Wikilinks use bare style: `[[Note Name]]`, not `[[masterclass/Note Name]]`.
- Tables: empty cells for "no value" rather than placeholder dashes.
- When in doubt, match the tone of the existing masterclass notes.

## Workflow Per Procedure

1. **Set up the folder.** Create `refactors/<ProcName>/` and prepare Original.sql and Refactored.sql per the git-history check above.
2. **Restate the tracking-sheet context** in Section 2. Pull the issue and refactor plan verbatim where useful.
3. **Pull cross-MFC Query Store evidence** into Section 3.1.
4. **Read the DDL closely.** Map the surface area. Document tables, indexes, callers in Section 1.
5. **Identify anti-patterns line by line** in Section 4.
6. **Articulate the principles** in Section 5. Cite existing masterclass notes; create new ones for net-new patterns.
7. **Write the refactor** in Refactored.sql with inline comments. Mirror the commented structure into Section 6.
8. **Document risk and rollback** in Section 7.
9. **Set up evidence placeholders** in Section 3.2 and Section 8 for my STATS IO/TIME captures.
10. **Document index recommendations and other open items** in Section 11.
11. **Capture** STATS IO/TIME and save the plan files.
12. **Once evidence is in,** populate Section 3.2, Section 8, Section 9 comparison table, and Section 10 validation checklist.
13. **Update the tracking sheet row** with the link to Analysis.md and the new status.

## Quality Bar

The pilot Analysis.md (`refactors/lsp_ShpGetOrdersForTopReadyToShipGroup/Analysis.md`) is the reference. Each subsequent analysis should match its depth, structure, and prose quality. If a procedure is small enough that the full template feels heavyweight, follow the template anyway. The consistency is more valuable than the brevity. The reader of these documents may be reviewing six of them in a sitting; they should not have to recalibrate per proc.

## Lessons Learned (cross-cutting)

Read `tasks/lessons.md` before each session. Add to it after each correction. The current entries cover: index choice verification, LEFT JOIN OR variants, LOB column predicates, apples-to-apples protocol, cold cache misrepresentation, 0-row test sets, near-empty driver states, OPTION clause statement-level scope, body diff against committed code, NoCount visibility, prose conventions, and version label ambiguity.
